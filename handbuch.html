GameEngine Benutzerhandbuch

Willkommen zur vollständigen Dokumentation der GameEngine - einer modernen, modularen Java-basierten Spiel-Engine auf Basis von JavaFX.

⏰ Zeitsteuerung & Aufgabenplanung

⚡
GameScheduler
Präzise Zeitsteuerung mit 50 TPS

Architecture

Tick-System: Der Scheduler verwendet ein präzises Tick-System mit 50 Ticks pro Sekunde (20ms pro Tick) für konsistente Game-Logic.

Aufgaben-Typen

Synchron (Game-Thread)
  • runRepeating() - Für Game-Logic
  • runLater() - Für verzögerte Aktionen
  • runNextTick() - Für sofortige nächste-Tick-Aktionen
Asynchron (Background)
  • runAsync() - Für langlaufende Aufgaben
  • ExecutorService - Thread-Pool Management
  • Non-blocking - Blockiert nicht die Game-Loop

Praxisbeispiele

// Game-Logic Loop (50 TPS) scheduler.runRepeating(this::updateGameLogic, 0, 1); // Langsame Updates (1 TPS) scheduler.runRepeating(this::updateSlowLogic, 0, 50); // Verzögerte Aktion scheduler.runLater(() -> { showNotification("Willkommen!"); }, 100); // 2 Sekunden warten (100 Ticks) // Asynchrone Aufgabe scheduler.runAsync(() -> { // Lade-Ressourcen im Hintergrund List<Texture> textures = loadTextures(); Platform.runLater(() -> { applyTextures(textures); }); });

Debug-Features

// Debug-Modus aktivieren GameScheduler.debug = true; GameScheduler.debugOnlyOnLag = true; // Nur bei Lag ausgeben GameScheduler.lagThresholdMillis = 25; // Lag-Schwelle GameScheduler.debugIntervalTicks = 50; // Ausgabe-Intervall
Performance-Monitoring: Der Scheduler überwacht automatisch die Performance und gibt Warnungen bei Lag aus.
📊
ScheduledTask
Verwaltete Aufgaben mit Lebenszyklus

Task-Management

// Task erstellen und verwalten ScheduledTask task = scheduler.runRepeating(() -> { updateParticles(); }, 0, 2); // Alle 2 Ticks // Task später abbrechen scheduler.runLater(() -> { task.cancel(); // Task stoppen }, 1000); // Nach 20 Sekunden

Task-Prioritäten

  • Sequence: Tasks werden in Reihenfolge ihrer Erstellung ausgeführt
  • NextRun: Tasks mit früherer nextRun-Zeit werden zuerst ausgeführt
  • Separation: Sync und Async Tasks werden in separaten Queues verwaltet

public class SimplePongGame extends GameEngine { private GraphicText scoreText; private int playerScore = 0, aiScore = 0; private double ballX = 400, ballY = 300; private double ballVX = 5, ballVY = 3; private double paddleY = 250; private double aiY = 250; @Override public void onStart(GraphicWindow window, GameScheduler scheduler) { setupGameGUI(window); setupControls(window); startGameLoop(scheduler); } private void setupGameGUI(GraphicWindow window) { window.main(main -> { // Score-Anzeige scoreText = main.addItemAndGet( GraphicText.builder() .text("0 : 0") .fontSize(32) .bold(true) .color(Color.WHITE), Alignment.of(0.5, 0.1) ); // Mittellinie for (int i = 0; i < 10; i++) { main.addItem( GraphicText.builder() .text("|") .fontSize(24) .color(Color.fromRGB(100, 100, 100)), Alignment.of(0.5, 0.15 + i * 0.08) ); } // Spieler-Paddle main.addItemAndGet( GraphicButton.builder() .text("") .backgroundColor(Color.WHITE) .size(15, 80) .build(), Position.of(50, 250) ); // AI-Paddle main.addItemAndGet( GraphicButton.builder() .text("") .backgroundColor(Color.WHITE) .size(15, 80) .build(), Position.of(735, 250) ); // Ball main.addItemAndGet( GraphicButton.builder() .text("") .backgroundColor(Color.WHITE) .size(15, 15) .roundedEdges(0.5) .build(), Position.of(400, 300) ); }); } private void setupControls(GraphicWindow window) { // Steuerung registrieren window.addKeyListener(Key.W, ActionType.PRESS, () -> movingUp = true); window.addKeyListener(Key.W, ActionType.RELEASE, () -> movingUp = false); window.addKeyListener(Key.S, ActionType.PRESS, () -> movingDown = true); window.addKeyListener(Key.S, ActionType.RELEASE, () -> movingDown = false); // Pause-Funktion window.addKeyListener(Key.P, ActionType.PRESS, this::togglePause); } private void startGameLoop(GameScheduler scheduler) { scheduler.runRepeating(this::updateGame, 0, 1); } private void updateGame() { if (isPaused) return; // Spieler-Bewegung if (movingUp && paddleY > 0) paddleY -= 8; if (movingDown && paddleY < 520) paddleY += 8; // AI-Bewegung double aiTarget = ballY - 40; if (aiY < aiTarget - 5) aiY += 6; if (aiY > aiTarget + 5) aiY -= 6; aiY = Math.max(0, Math.min(520, aiY)); // Ball-Bewegung ballX += ballVX; ballY += ballVY; // Ball-Kollision mit Wänden if (ballY <= 0 || ballY >= 585) { ballVY = -ballVY; } // Ball-Kollision mit Paddles if (ballX <= 65 && ballX >= 50 && ballY >= paddleY && ballY <= paddleY + 80) { ballVX = Math.abs(ballVX); } if (ballX >= 735 && ballX <= 750 && ballY >= aiY && ballY <= aiY + 80) { ballVX = -Math.abs(ballVX); } // Punkte vergeben if (ballX < 0) { aiScore++; resetBall(); } if (ballX > 800) { playerScore++; resetBall(); } // GUI aktualisieren updateGUI(); } private void updateGUI() { scoreText.text(playerScore + " : " + aiScore); // Paddle- und Ball-Positionen aktualisieren... } private void resetBall() { ballX = 400; ballY = 300; ballVX = (Math.random() > 0.5 ? 5 : -5); ballVY = (Math.random() - 0.5) * 6; } @Override public void tickLoop() { // Zusätzliche Game-Logic bei Bedarf } @Override public void secondLoop() { // Langsame Updates (z.B. Partikel-Effekte) } @Override public String getName() { return "Simple Pong"; } }

✨ Best Practices & Entwicklungstipps

⚡
Performance
Optimierungstipps für maximale Performance

Game-Loop Optimierung

  • Halte tickLoop() kurz: Nur essentielle Game-Logic, keine langen Operationen
  • Nutze runAsync() für Heavy Tasks: Datei-Laden, Netzwerk-Operationen
  • Dirty-Flag-Pattern: GUI-Updates nur bei Änderungen durchführen
  • Object Pooling: Wiederverwendung von Objekten statt ständiger Neuerstellung

Memory Management

// ❌ Schlecht: Objekte werden ständig neu erstellt public void updateParticles() { for (int i = 0; i < 100; i++) { Particle p = new Particle(); // Memory-Leak! particles.add(p); } } // ✅ Gut: Object Pooling verwenden private Queue<Particle> particlePool = new ArrayDeque<>(); public void updateParticles() { Particle p = particlePool.poll(); if (p == null) p = new Particle(); // Particle verwenden... particlePool.offer(p); // Zurück zum Pool }
Performance-Tipp: Verwende das eingebaute Caching-System für Texturen und andere Ressourcen!

Robuste Fehlerbehandlung und Debugging

Debugging

// Debug-Helper Klasse public class GameDebugger { private static final boolean DEBUG = true; public static void log(String message) { if (DEBUG) { System.out.println("[DEBUG] " + message); } } public static void assertNonNull(Object obj, String name) { if (obj == null) { throw new IllegalArgumentException(name + " cannot be null!"); } } public static void measureTime(String operation, Runnable runnable) { long start = System.nanoTime(); runnable.run(); long duration = System.nanoTime() - start; log(operation + " took " + (duration / 1_000_000) + " ms"); } }

Architektur
Strukturierte Code-Organisation

Package-Struktur

game/ ├── engine/ // Engine-Kern │ ├── GameEngine.java │ ├── GameScheduler.java │ └── GraphicWindow.java ├── gui/ // GUI-System │ ├── components/ │ ├── overlays/ │ └── themes/ ├── game/ // Spiel-spezifisch │ ├── entities/ │ ├── systems/ │ └── logic/ ├── input/ // Input-System ├── resources/ // Ressourcen └── utils/ // Hilfsklassen

Design Patterns

Empfohlene Patterns
  • Builder Pattern: Für GUI-Komponenten
  • Observer Pattern: Für Events
  • Factory Pattern: Für Entity-Erstellung
  • State Pattern: Für Game-States
Anti-Patterns
  • God Classes vermeiden
  • Keine statischen Dependencies
  • Keine Hardcoded Values
  • Keine Circular Dependencies
Architektur-Tipp: Halte die Engine sauber von Spiel-spezifischem Code getrennt!
